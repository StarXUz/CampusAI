# Campus Service Manager AI 项目实施过程中主要问题与处理情况

## 1. 文档说明

本文档用于单独记录项目开发与联调过程中出现的主要问题、原因定位和处理结果，可直接作为课程答辩中的“项目难点与问题解决”说明材料。

---

## 2. 三端登录态串扰问题

### 2.1 问题现象

- 管理端、学生端、超级端同时打开时，刷新页面会出现登录态串扰
- 超级端登录后可能跳回后台登录页
- 不同端口页面之间存在 Cookie 干扰

### 2.2 原因定位

项目虽然使用了不同端口，但早期仍存在相同 host 访问的情况，浏览器会复用同一组域名下的 Cookie，从而导致三端会话边界不清晰。

### 2.3 处理结果

- 管理端固定为 `127.0.0.1:8091`
- 学生端固定为 `localhost:8092`
- 超级端固定为 `super.localhost:8093`
- 同步修正 Spring Security 登录成功跳转和跨端重定向逻辑

当前三端已能够并行打开，刷新后不再出现串端问题。

## 3. 超级端页面导航异常问题

### 3.1 问题现象

- 超级端首页部分入口点击后无法跳转
- 超级端内“学生端用户管理”“管理端账号管理”等入口在主页面中无法正确联动

### 3.2 原因定位

超级端模板缺少完整的三端 host / port 配置注入，部分跳转仍依赖旧地址或当前来源地址，导致 URL 计算不稳定。

### 3.3 处理结果

- 在页面渲染阶段补齐超级端、学生端、管理端的完整入口配置
- 将超级端侧边栏、卡片入口、页头跳转统一改为基于固定配置生成
- 补充前端导航绑定逻辑，实现主页面内部切换与 URL 同步

当前超级端主页导航已恢复正常。

## 4. 默认账号密码异常问题

### 4.1 问题现象

- 后台管理端默认账号登录失败
- 超级端默认账号登录失败
- 演示账号与数据库中实际密码不一致

### 4.2 原因定位

历史调试过程中，部分账号密码经过多轮修改，数据库中的实际加密结果与答辩使用的默认账号说明不一致。

### 4.3 处理结果

- 统一校正后台管理端和超级端默认账号密码
- 重新核对数据库中的演示账号状态
- 固定当前答辩账号口径：
  - 后台管理端：`merchant_north / 123456`
  - 超级管理端：`platform_super / 123456`

当前默认演示账号已恢复可用状态。

## 5. AI 聊天输入框残留问题

### 5.1 问题现象

- 学生端 AI 聊天发送消息后，输入框内容未被及时清空
- 高频连续发送时，容易造成重复内容残留

### 5.2 原因定位

前端消息发送逻辑与输入框状态清理时机不够稳定，部分异步请求完成前后存在状态未重置的问题。

### 5.3 处理结果

- 调整消息发送后的输入框重置逻辑
- 对发送成功后的界面状态清理做了加强处理
- 保证连续发送场景下输入框内容可及时清空

当前 AI 输入框残留问题已完成修复。

---

## 6. 编辑表单与页面布局冲突问题

### 6.1 问题现象

- 多个管理页面中，点击“编辑”后会在当前页面右侧或下方直接展开表单
- 页面被大面积编辑区挤压，信息阅读和操作体验较差
- 个别模块已经改成弹窗，但原内联表单仍残留在页面中

### 6.2 原因定位

项目前期为了快速完成 CRUD，采用了“列表 + 内联编辑区”的实现方式。后续在升级为弹窗编辑时，部分模块的旧表单容器未完全清理，且个别 `id` 绑定到了错误的面板节点。

### 6.3 处理结果

- 将管理端、超级端、学生端的主要新增 / 编辑功能统一改为弹窗编辑
- 清理页面内遗留的重复表单区域
- 修正活动表单、菜品表单等模块中错误的面板绑定关系

当前主要编辑操作已统一为弹窗模式，页面结构更加稳定。

## 7. 活动 PDF 导出失败问题

### 7.1 问题现象

- 点击“导出 PDF”后浏览器跳转到一个无法正常打开的地址
- 页面出现“当前页面已经永久转移”等异常提示
- 后续改为下载方式后，后端仍出现 PDF 生成失败

### 7.2 原因定位

该问题经历了两个阶段：

1. 前端直接以超链接方式访问 PDF 接口，浏览器将其当作页面跳转处理，导致用户体验异常。
2. 后端最初依赖 JasperReports 动态编译 JRXML 模板生成 PDF，模板中存在 `pageFooter` 结构不符合约束的问题，导致运行时报错。

### 7.3 处理结果

- 前端改为 `blob` 下载方式，不再整页跳转
- 后端放弃脆弱的 Jasper 动态模板生成方案，改为使用 OpenPDF 原生生成活动报名报表
- 补充中文字体文件兜底逻辑，避免 PDF 中文显示异常

当前活动报名 PDF 已可正常下载，且数据与数据库保持一致。

## 8. 活动静态页报名人数不一致问题

### 8.1 问题现象

- 学生端报名成功后，活动静态页仍显示 `0` 人报名
- 同一活动的 PDF 报表中报名人数是正确的

### 8.2 原因定位

手动点击“生成静态页”时，旧逻辑只传递了活动基础信息，没有重新读取最新报名人数，导致静态页生成时使用的是不完整数据。

### 8.3 处理结果

- 将手动生成静态页逻辑统一切换到服务层集中处理
- 生成前重新查询活动当前报名人数
- 保证静态页、活动详情数据、PDF 报表三者统计口径一致

当前活动静态页报名人数已与数据库实时一致。

## 9. 超级端入口不便直接启动问题

### 9.1 问题现象

- 启动日志中超级端地址显示为 `http://[::1]:8093/login/super`
- 在 IDEA 终端中不易直接点击打开

### 9.2 原因定位

早期超级端入口使用 IPv6 形式输出，虽然服务本身能够正常启动，但在常用开发环境下不利于直接打开和演示。

### 9.3 处理结果

- 将超级端访问地址统一调整为 `http://super.localhost:8093/login/super`
- 保持与另外两端不同 host，继续满足登录态隔离要求

当前三端入口均已具备较好的启动展示与点击访问体验。

## 10. 月度运营报表多出口同源问题

### 10.1 问题现象

- 页面月度运营报表预览已经存在
- 但缺少 PDF 导出能力
- 若分别维护页面、Excel、PDF 三份逻辑，容易产生数据口径不一致

### 10.2 原因定位

报表预览和 Excel 导出最初已具备实现，但 PDF 能力缺失，且各出口若独立拼装数据，会增加维护复杂度。

### 10.3 处理结果

- 增加月度运营报表 PDF 导出
- 将页面预览、Excel 导出、PDF 导出统一到同一份数据库查询结果
- 保证商家、营收、订单量、退款率等核心字段口径一致

当前月报三种输出方式已实现同源。

---

## 11. 座位导入演示数据不足问题

### 11.1 问题现象

- 原始座位导入模板数据量偏少
- 难以支撑自习室预约模块的完整批量导入演示

### 11.2 原因定位

模板最初更偏向字段示例，缺少与当前数据库规模相匹配的完整演示数据。

### 11.3 处理结果

- 基于项目当前数据库备份中的自习室与座位数据重新生成模板
- 将首个工作表扩充为可直接导入的完整数据
- 当前模板已覆盖 `3` 个自习室、共 `660` 个座位

当前座位模板已满足批量导入与课堂演示需求。

## 12. ZooKeeper 未启用的说明问题

### 12.1 问题现象

- 后台运行状态页面展示了 `Redis / Quartz / ZooKeeper`
- 其中 ZooKeeper 显示“未启用”，容易被误解为系统异常

### 12.2 原因定位

ZooKeeper 在当前项目中承担的是扩展型配置中心展示角色，而不是核心业务前置依赖。如果未说明清楚，容易让评审误以为系统运行不完整。

### 12.3 处理结果

- 明确项目当前默认使用本地配置
- 明确 ZooKeeper 未启用不会影响登录、下单、预约、活动、报表、AI 等主体业务
- 将其答辩口径统一为“扩展能力保留项”

当前该问题已转化为可清晰说明的系统架构设计点。

---

## 13. 问题处理总结

从本项目实施过程看，主要问题集中在以下几个方面：

- 多端登录态隔离与页面跳转稳定性
- 前端交互方式统一与页面结构收口
- 报表导出链路稳定性
- 静态页与数据库统计一致性
- 演示环境入口、账号和模板材料准备

上述问题均已完成定位与针对性处理，项目当前已具备较稳定的答辩演示条件。
